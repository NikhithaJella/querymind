# QueryMind — System Architecture

## Overview

QueryMind is a full-stack Text-to-SQL analytics application. It has two independently deployed services:

- **Backend** — Python / FastAPI, deployed on Render.com
- **Frontend** — React 19 / TypeScript / Vite, deployed on Vercel

```
Browser
  │
  ├─ React Frontend (Vercel)
  │    ├─ Sidebar.tsx       — CSV upload, dataset list, query history
  │    ├─ ChatPanel.tsx     — Question input, message thread
  │    └─ InsightsPanel.tsx — Table / Chart / Insights tabs
  │
  │  HTTP REST (JSON)
  │
  └─ FastAPI Backend (Render)
       ├─ POST /upload      — load CSV into DuckDB, run profiler
       ├─ POST /query       — generate SQL, execute, return insights
       ├─ GET  /history     — retrieve last 5 turns
       └─ DELETE /history   — clear session
```

## Backend

### FastAPI (`backend/main.py`)

The entry point. Defines four HTTP endpoints. Each request carries a `session_id` (a UUID generated in the browser and stored in `localStorage`). The backend uses this ID to look up the correct in-memory DuckDB connection, conversation history, and query cache for that user.

CORS is open to all origins — intentional for a demo deployment where the frontend and backend are on separate domains.

### DuckDB Session Manager (`backend/db_duckdb.py`)

DuckDB is an in-process SQL analytics engine. It runs entirely in memory with no external database server required. Each session gets its own `duckdb.Connection` stored in a Python dict keyed by `session_id`.

When a CSV is uploaded:
1. File bytes are written to a temporary file
2. DuckDB reads it: `CREATE TABLE name AS SELECT * FROM read_csv_auto(path, sample_size=2000)`
3. Column types (INTEGER, DOUBLE, VARCHAR, DATE) are auto-detected
4. The connection is stored in `_sessions[session_id]`

When a query runs, `execute_query()` calls `conn.execute(sql).fetchmany(500)` and returns rows + column names.

`get_schema_context()` builds a prompt-ready string with table name, column types, and 3 sample rows — this is injected into every SQL generation prompt.

### LLM SQL Generation (`backend/agent/sql_agent.py`)

SQL is generated by calling the Groq API with a structured prompt. The model used is `llama-3.3-70b-versatile`.

The prompt is assembled from:
- A system instruction (DuckDB dialect, SELECT-only, use exact column names)
- The table schema and 3 sample rows from `get_schema_context()`
- The last 5 conversation turns from `memory.py`
- The current user question
- On retry: the failed SQL and the error message

The model returns SQL inside triple-backtick fences. `_extract_sql()` parses it out with regex. If extraction fails, a secondary regex scans for any bare `SELECT` statement.

### SQL Validation (`backend/security.py`)

Every generated SQL string passes through `validate_sql()` before execution:

1. `sqlglot.parse(sql)` builds an AST. Each top-level statement is checked — if any is not `exp.Select`, an `HTTPException(403)` is raised immediately.
2. If sqlglot raises a parse error, a regex fallback checks for blocked keywords: `DROP`, `DELETE`, `INSERT`, `UPDATE`, `CREATE`, `ALTER`, `TRUNCATE`, `REPLACE`, `MERGE`, `GRANT`, `REVOKE`.

This means only `SELECT` statements ever reach DuckDB.

### Query Repair Loop (`backend/agent/sql_agent.py` — `run_agent()`)

```
Attempt 1
  └─ generate SQL → validate → execute
       ├─ success → return result, confidence="High"
       └─ error → append (failed_sql, error) to retry_history

Attempt 2
  └─ generate SQL with error context → validate → execute
       ├─ success → return result, confidence="Medium"
       └─ error → append to retry_history

Attempt 3
  └─ generate SQL with full history → validate → execute
       ├─ success → return result, confidence="Low"
       └─ error → raise HTTPException(422)
```

The retry history (failed SQL + error pairs) is included verbatim in the next prompt so the LLM can see exactly what went wrong.

### Insights Pipeline

After a query succeeds, three modules run in sequence:

**Anomaly detection** (`backend/insights/anomaly.py`)
- For each numeric column with ≥4 values: compute Q1, Q3, IQR
- Flag values outside `Q1 − 1.5×IQR` or `Q3 + 1.5×IQR` with z-score and row label
- Flag columns with >10% nulls
- Return up to 5 findings

**Chart selection** (`backend/insights/chart.py`)
- Identify string columns (candidates for x-axis) and numeric columns (y-axis)
- If x-axis column name contains a date keyword → line chart
- If ≤6 unique x values → pie chart
- Otherwise → bar chart
- Return first 20 rows formatted for Recharts

**Narrative generation** (`backend/insights/narrative.py`)
- Sends question + SQL + first 5 result rows + optional web context to Groq
- Asks for a 2–3 sentence business narrative and 2 follow-up questions
- Returns `{narrative, suggestions}`

### Real-time Web RAG (`backend/retrieval/web_rag.py`)

If the user's question contains keywords like `market`, `trend`, `benchmark`, `industry`, `compare`, `growth`, or if the query returns zero rows, the app calls Tavily's search API. The top 3 results (title + 600-char snippet + URL) are injected into the narrative prompt as external context. This is the Retrieval-Augmented Generation (RAG) component.

### Query Cache (`backend/cache.py`)

Per session, answered questions are stored as `(normalized_question, response, timestamp)`. Normalization: lowercase + collapse whitespace. On a new question, the cache is checked first. Cache entries expire after 1 hour (TTL = 3600s). A cache hit skips all AI and database calls entirely.

### Conversation Memory (`backend/agent/memory.py`)

A `deque(maxlen=5)` per session stores `{question, sql, narrative}` for each completed turn. These are formatted and injected into every SQL generation prompt so follow-up questions resolve correctly (e.g., "now filter that by holiday weeks" knows what "that" refers to).

## Frontend

### State Orchestration (`src/routes/index.tsx`)

Holds all top-level state: `sessionId`, `tables[]`, `messages[]`, `latestResponse`. On mount, reads or creates a UUID from `localStorage`. Handles file upload → `handleUploaded`, question submit → `handleSubmit`, and session clear → `handleClearHistory`.

### Chat Interface (`src/components/ChatPanel.tsx`)

Renders the message thread. User messages appear as bubbles. Assistant responses show the Data Passport card (on upload) or query result summary (on question). While a query runs, a 6-step animated thinking timeline plays. Suggested follow-up questions appear as clickable chips.

### Results Panel (`src/components/InsightsPanel.tsx`)

Three tabs:
- **Table** — paginated results, CSV download
- **Chart** — Recharts Bar/Line/Pie, client-side sort and filter controls, PNG download
- **Insights** — AI narrative, anomaly list, confidence badge, retry count

### API Client (`src/lib/api.ts`)

All backend calls are centralized here. TypeScript interfaces define the full shape of `UploadResponse` and `QueryResponse`. `getSessionId()` creates or reads the UUID from `localStorage`.

### HTML Report Export (`src/lib/report.ts`)

`downloadReport()` iterates over assistant messages with query results and generates a self-contained HTML file with inline Chart.js. No server call — runs entirely in the browser. The file can be opened offline and shared without dependencies.

## Data Flow Summary

```
CSV upload
  → DuckDB loads table (db_duckdb.py)
  → Profiler runs column stats + Groq classification (profiler.py)
  → Data Passport returned to frontend

Question asked
  → Cache check (cache.py)
  → Web search if needed (web_rag.py)
  → Schema + history assembled (db_duckdb.py, memory.py)
  → Groq generates SQL (sql_agent.py)
  → SQL validated (security.py)
  → DuckDB executes SQL (db_duckdb.py)
  → Anomalies detected (anomaly.py)
  → Chart config selected (chart.py)
  → Groq generates narrative (narrative.py)
  → Response cached (cache.py)
  → Full response returned to frontend
```

## Deployment

| Service | Platform | Config |
|---|---|---|
| Backend | Render.com | `render.yaml` |
| Frontend | Vercel | auto-detected Vite project |

The backend has no persistent disk — DuckDB sessions live in process memory and are lost on restart. This is an accepted constraint for a demo deployment.
